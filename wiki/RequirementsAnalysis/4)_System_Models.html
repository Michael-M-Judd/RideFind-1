4) System Models¶

4.1 Use Case Models¶

Name: RegisterNewUser
Actor: USER
Entry Conditions: Application is running.
USER is not registered.
Flow of Events: 1.USER initiates new profile function.
2.SYSTEM presents user with RegistrationForm
3.USER information is stored in the database
Exit Conditions: USER is now signed in.
Name: SignIn
Actor: USER
Entry Conditions: Application is running.
USER is not signed in.
Flow of Events: 1.SYSTEM presents user with the sign in page
2.USER inputs username and password
3.SYSTEM verifies user information
Exit Conditions: USER is now signed in.
Name: CreateNewRide
Actor: DRIVER
Entry Conditions: Application is running.
USER is signed in.
Flow of Events: 1.USER initiates ride offer function.
2.SYSTEM presents user with OfferForm
3.Ride information is stored in the database
4.SYSTEM adds the ride listing to the main page
5.SYSTEM confirms the ride offer with the USER
Exit Conditions: The ride listing is now updated to include the new ride offer.
Name: EmptyCar
Actor: SYSTEM
Entry Conditions: Application is running.
Flow of Events: 1.SYSTEM uses a timer to check if the ride listing has expired
2.SYSTEM checks for any booked spots
3.SYSTEM deletes the ride listing if all the spots are vacant
Exit Conditions: SYSTEM updates the main page.
Name: BookARide
Actor: PASSENGER
Entry Conditions: Application is running.
USER is signed in.
Flow of Events: 1.USER selects a ride listing from the main page
2.SYSTEM confirms the ride booking with the USER
3.SYSTEM sends the DRIVER an email confirmation
Exit Conditions: SYSTEM updates the ride listing showing one less spot in the car.
Name: FullCar
Actor: PASSENGER
Entry Conditions: Application is running.
USER is signed in.
Flow of Events: USER selects a ride listing from the main page
SYSTEM presents USER with a message indicating that the car is full
Exit Conditions: SYSTEM removes the ride listing from the main page.
Name: ReviewDriver
Actor: USER
Entry Conditions: Application is running.
USER is signed in.
Flow of Events: 1.SYSTEM uses a timer to indicate that a ride booking has expired
2.SYSTEM presents USER with previous/completed rides
3.USER submits a review
4.Review information is stored in the database
Exit Conditions: SYSTEM updates the DRIVER profile.
Name: EmailUser
Actor: USER
Entry Conditions: Application is running.
USER is signed in.
Flow of Events: 1.USER initiates the mailing function
2.SYSTEM presents USER with ComposeEmail page
3.USER composes and sends a message
Exit Conditions: SYSTEM presents USER with email confirmation page.
Name: FindARide
Actor: PASSENGER
Entry Conditions: Application is running.
USER is signed in.
Flow of Events: USER refines displayed ride listing using the checklist
Exit Conditions: SYSTEM presents USER with an updated ride listing.

4.2 User Interface - Navigational Paths and Screen Mock-ups¶


Figure 1 Main Screen

Menu Bar:¶

  • File
    o Log Out... (Ctrl+L)
    o Exit... (Ctrl+Q)
  • Edit
    o Undo (Ctrl+Z)
    o Redo (Ctrl+Y)
    o Search (Ctrl+L)
  • View
    o Zoom In (Ctrl++)
    o Zoom Out (Ctrl+-)
  • Help
    o About (F1)
    o Instructions (F2)

Secondary Menu Bar:¶

  • The second bar below the main bar includes some of the important features of the application.
    o Create Your Ride Here Button: brings the user to a screen to create their ride listing
    o Commute from: field where user can type where they want to start their trip
    o Commute to: field where use can type their destination.
    o Date: allows user to select the date of departure.
    o Search button: once the 3 fields are filled will define the results for ride listings and only
    show the ones that match the search.
    o Bell button: displays how many notifications the user has, and if clicked will bring the
    user to the window showing their recent interactions.
    o Envelope button: displays how many pending messages the user has, if clicked will bring
    the user to the messaging feature.
    o Username button: shows the users name. If clicked will bring the user to their own
    profile.

Left Side Menu Bar:¶

  • MAIN is the category listing the core functions of the program
    o Browse button brings the user back to the listings page
    o My Rides button brings the user to their rides they have agreed to take and have
    previously taken
    o My Profile is second button that brings the user to their profile
  • YOUR LISTINGS is the category to manage their own rides they have created
    o Add Ride button brings the user to a screen to create their ride listing
    o Manage Ride button brings the user to a screen to edit their own rides
  • SEARCH is the category to refine their search by an array of options
    o Sort By is a drop down menu where the user can choose an option to sort the listings by
    including price, driver name, destination, departure location, seats, ending soonest and
    ending latest.
    o Free Ride: refines the listings to only rides that cost 0.
    o Two Seats: refines the listings to at least two seat listings.
    o This Week: refines the listings to a maximum of ending within the week of current time.
    o Profile Picture: text underneath "Review your last driver." Will bring the user to a page
    where they can rate their driver out of 5 stars and give a comment for their profile.
    Find Your Ride Table:
  • Find your ride table is the main function of the program. It shows a list of the users who have
    posted a ride listing. The listing includes the drivers name, destination, departure location, date
    of departure, time of departure, seats available, message button, and price. The listings are
    organized by default by listings ending soonest.
    o Message button: opens a window where you can send a message to the listing creator.
    o Listing button: The whole listing itself is a clickable button which will open a new
    window where you can confirm to take the ride.
    o Keyboard only features include:
    ? DOWN and UP to scroll through listings.


Figure 2 Startup Page

Startup Page:¶

  • This is the first page the user will see opening the program for the first time.
  • Includes information about RideFind on the left.
  • List of required information the user must type in on the right hand side:
    o Name: First and last name
    o Student E-Mail: Email ending with a university domain
    o Choose a Password: password, minimum length 4 characters
    o Confirm Password: must be the same string as the Choose a Password field
    o Birthday: dropdown selection of month number, day number, and year number
    o Gender: field where user can type their gender
    o Mobile Phone: phone number
    o Location: where user is located
    o Car Model: user can type their car model if applicable.
    o Profile Picture: user can choose an image file from their computer to create a profile
    picture
    o Sign Up button: requires all but Gender, Mobile Phone and Car Model to be filled out, it
    will then put the user into the database.
    o Sign In button: will close the current screen and open the Sign In screen.


Figure 3 Sign in page

Sign In Page:¶

  • After opening the application more than once, this will be the first page the user sees.
  • This is where the user will log into the program using his information they signed up with.
    o Student E-Mail: User enters their student email
    o Password: User enters their password
    o Continue Button: If valid login is in database, the main screen will appear, if not, it will
    prompt the user to try again
  • Keyboard only features include:
    o ENTER: if valid login is in database, the main screen will appear, if not, it will prompt the
    user to try again.
  • The bottom text is an executive summary of our program.
